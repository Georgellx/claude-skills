# 浏览器自动化实践经验（browser-automation-notes）

第 5 步可选：当连接了 Claude in Chrome 这类浏览器自动化工具时，如何高效、低风险地协助拉取职位和投递。经验来自真实跑通 51job 的实践。

## 核心原则：读写分离

招聘网站普遍有反爬。实测的关键结论：

- **"读"几乎不受限**：用页面内 JS（`javascript_tool`）一次能提取整页 20+ 个职位的标题/薪资/地区/公司/链接；`get_page_text` 能一次拿整页 JD。批量读没问题。
- **"写"才是瓶颈**：投递动作背后有加密签名和风控埋点。所以——**读批量化、写拟人化**：拉取/抓取尽管批量跑，投递则放慢、拟人点击，别把低风险的读拖进高风险的写通道。

## 投递的几个实操要点

1. **优先在列表页点"投递"按钮**：比"进详情页再投"省一半操作，且实测有效（会弹"投递成功"）。
2. **拿不到真实职位链接**：很多站点（如 51job）用 Vue/JS 动态 `window.open` 打开详情，DOM 上 href 是 `javascript:;`。解决：临时桩掉 `window.open` 捕获目标 URL。带追踪参数的完整 URL 可能被安全层拦截，取 `origin+pathname` 即可。
3. **投递是否成功别只看弹窗**：曾出现弹窗没出来、需二次点击的情况。**投完回"我的申请/投递记录"页用客观数据核对到账**，这才算数。
4. **去重**：投之前先比对投递记录，避免重复投同一家。
5. **节奏**：批量投放慢，每投几个停一下；遇到滑块/人机验证，交给用户手动过，不要硬刚。

## 铁律仍然适用

即使有自动化工具，**投递前仍要把清单摆出来让用户确认投哪些**（SKILL.md 铁律 2）。自动化只加速"拉取和点击"，不改变"投递决定权在用户"这条线。验证码、账号登录、留联系方式等动作交给用户本人。

## 连接排查

如果 `list_connected_browsers` 返回空：扩展能聊天 ≠ 已注册为可控浏览器，桥接是会断会重连的 WebSocket，有延迟。等一下/重连通常即可，多数不是配置问题。需要深查时看桌面端日志 `%APPDATA%\Claude\logs\main.log`。
